## The problem

Ask an AI agent: *"Which customers that churned last quarter had declining usage AND open support tickets?"*

It can't answer. The data lives in Stripe, HubSpot, and Intercom -- each behind a separate API. You can't `JOIN` across REST endpoints. You can't `GROUP BY` across two SaaS tools.

With Dinobase, the agent writes one SQL query and gets the answer:

```sql
SELECT s.name, s.email, h.company, d.amount, d.dealstage
FROM stripe.customers s
JOIN hubspot.contacts h ON s.email = h.email
JOIN hubspot.deals d ON h.id = d.contact_id
WHERE d.dealstage = 'closedwon'
ORDER BY d.amount DESC
```

## How it works

```
                Agent (Claude, GPT, etc.)
                          |
                +---------+---------+
                |                   |
           MCP Server             CLI
           (tool calls)       (bash commands)
                |                   |
                +---------+---------+
                          |
                    Query Engine
                    (DuckDB SQL)
                          |
             +------------+------------+
             |            |            |
       stripe.*     hubspot.*    analytics.*
       (synced)     (synced)    (parquet views)
```

| Source type | How it works | Data location |
|------------|-------------|---------------|
| API sources (100+ SaaS connectors, databases) | dlt syncs to parquet | `~/.dinobase/data/` |
| File sources (parquet, CSV, S3) | DuckDB reads via views | Your storage -- nothing copied |

Each source becomes a schema (`stripe.*`, `hubspot.*`). Cross-source joins work via shared columns like email. The `.duckdb` file stores only metadata -- data lives in parquet.
